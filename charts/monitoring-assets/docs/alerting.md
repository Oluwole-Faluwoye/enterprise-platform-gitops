# Enterprise Alerting

## Overview

Alerting is one of the core pillars of an enterprise observability platform.

While dashboards provide engineers with visibility into the current state of a system, alerts proactively notify engineers when predefined conditions indicate that a service, application, or infrastructure component requires attention.

Without alerting, engineers must constantly monitor dashboards to identify problems. This approach does not scale and significantly increases Mean Time To Detection (MTTD).

The Enterprise Platform uses Prometheus Alerting Rules together with Alertmanager to automatically detect failures, route notifications, and reduce operational response time.

---

# Objectives

The alerting architecture has been designed to achieve the following objectives:

- Detect failures automatically
- Notify engineers before users are impacted
- Reduce Mean Time To Detection (MTTD)
- Reduce Mean Time To Recovery (MTTR)
- Standardize operational monitoring
- Eliminate manual system monitoring
- Integrate with enterprise notification channels
- Support scalable platform operations

---

# What is Alerting?

Alerting is the process of continuously evaluating system metrics against predefined conditions.

When a condition remains true for a specified duration, Prometheus creates an alert and forwards it to Alertmanager.

Alertmanager then determines:

- Who should receive the alert
- Where the alert should be delivered
- How alerts should be grouped
- Whether duplicate alerts should be suppressed
- Whether notifications should be delayed or repeated

This enables engineers to respond quickly to operational issues without constantly watching dashboards.

---

# Why Alerting Matters

Dashboards are reactive.

Alerts are proactive.

Dashboards answer questions such as:

- Is CPU utilization increasing?
- How many Pods are running?
- What is the request latency?

Alerts answer different questions:

- CPU utilization has exceeded the threshold.
- A Kubernetes node is unavailable.
- The application is returning HTTP 500 errors.
- A deployment has failed.

Without alerts, engineers only discover issues after manually inspecting dashboards.

In enterprise environments, systems must notify engineers automatically.

# Alert Lifecycle

The Enterprise Platform follows the alert lifecycle shown below.

```
Application

↓

Micrometer Metrics

↓

Prometheus

↓

Recording Rules

↓

Alert Rules

↓

Alertmanager

↓

Slack / Email

↓

Platform Engineer
```

Each component has a specific responsibility.

## Application

Applications expose metrics through Spring Boot Actuator and Micrometer.

Examples include:

- HTTP requests
- Response time
- JVM memory
- Thread usage
- Database connections

---

## Prometheus

Prometheus scrapes metrics from applications and Kubernetes components at regular intervals.

It stores these metrics as time series.

---

## Recording Rules

Recording Rules optimize expensive PromQL expressions.

Instead of recalculating complex queries repeatedly, Prometheus stores precomputed metrics.

These optimized metrics become the foundation for dashboards and alert rules.

---

## Alert Rules

Alert Rules continuously evaluate metrics and determine whether an operational condition requires attention.

Examples include:

- High CPU usage
- High memory utilization
- Pod restart loops
- Deployment failures
- Increased HTTP error rate

Each rule contains:

- Expression
- Threshold
- Evaluation duration
- Severity
- Human-readable annotations

---

## Alertmanager

Alertmanager receives alerts generated by Prometheus.

Its responsibilities include:

- Deduplicating alerts
- Grouping related alerts
- Routing alerts
- Sending notifications
- Silencing alerts
- Managing notification timing

Alertmanager ensures engineers receive meaningful notifications rather than hundreds of duplicate alerts.

# Severity Levels

Not every issue requires the same operational response.

The Enterprise Platform classifies alerts into severity levels.

---

## Info

Information only.

Examples:

- Backup completed
- New deployment
- New node joined cluster

No immediate action is required.

---

## Warning

Warning alerts indicate conditions that may become operational problems if ignored.

Examples include:

- CPU above 80%
- Memory above 85%
- Disk usage above 80%
- Increased application latency

Warning alerts allow engineers to investigate before service degradation occurs.

---

## Critical

Critical alerts indicate immediate operational issues that require prompt action.

Examples include:

- Node unavailable
- Deployment unavailable
- Application unreachable
- Persistent HTTP 500 errors
- Disk almost full
- Kubernetes control plane unavailable

Critical alerts should trigger immediate investigation and remediation.

---

# Alert Categories

To improve ownership and maintainability, alert rules are grouped into logical domains.

---

## Infrastructure Alerts

Infrastructure alerts monitor the underlying compute resources supporting the Kubernetes platform.

Examples include:

- CPU utilization
- Memory utilization
- Disk usage
- Filesystem capacity
- Node availability

These alerts are primarily used by Platform Engineers and Infrastructure Engineers.

---

## Kubernetes Alerts

Kubernetes alerts monitor the health of workloads running inside the cluster.

Examples include:

- Pending Pods
- CrashLoopBackOff
- Deployment failures
- StatefulSet failures
- DaemonSet failures
- PVC capacity

These alerts ensure the Kubernetes platform remains healthy.

---

## Application Alerts

Application alerts monitor business services running on Kubernetes.

Examples include:

- HTTP error rate
- Request latency
- Application availability
- JVM memory
- JVM garbage collection
- Authentication failures

These alerts focus on the health and performance of the application itself rather than the underlying infrastructure.

# Best Practices

The Enterprise Platform follows several alerting best practices.

---

## Alerts Must Be Actionable

Every alert should require an action.

If an engineer cannot determine what action to take after receiving an alert, the alert should be redesigned.

Avoid alerts that provide information without operational value.

---

## Avoid Alert Fatigue

Too many alerts reduce operational effectiveness.

Engineers quickly begin ignoring notifications when most alerts are not important.

To reduce alert fatigue:

- Use appropriate thresholds
- Configure evaluation periods
- Group related alerts
- Remove noisy alerts

Quality is more important than quantity.

---

## Use Recording Rules

Alert rules should reference Recording Rules whenever possible.

Instead of repeatedly executing expensive PromQL expressions, alerts should evaluate precomputed metrics.

Benefits include:

- Faster evaluations
- Lower Prometheus CPU utilization
- Consistent calculations
- Easier maintenance

---

## Group Alerts by Domain

Infrastructure, Kubernetes, and Application alerts should remain in separate PrometheusRule resources.

Benefits include:

- Clear ownership
- Easier reviews
- Simpler troubleshooting
- Better repository organization

---

## Use Multiple Severity Levels

Not every issue is critical.

Implementing warning and critical thresholds provides engineers with sufficient time to investigate potential issues before they become service outages.

---

## Include Useful Annotations

Every alert should provide enough information for an engineer to understand the issue immediately.

Recommended annotations include:

- Summary
- Description
- Runbook URL
- Dashboard URL
- Suggested remediation

Well-written alerts significantly reduce Mean Time To Recovery.

# Validation

After Argo CD synchronizes the Monitoring Assets chart, verify that all alert rules have been deployed successfully.

List all PrometheusRule resources.

```bash
kubectl get prometheusrules -n monitoring
```

Expected resources include:

- enterprise-recording-rules
- infrastructure-alerts
- kubernetes-alerts
- application-alerts

Inspect each alert definition.

Infrastructure alerts:

```bash
kubectl describe prometheusrule infrastructure-alerts -n monitoring
```

Kubernetes alerts:

```bash
kubectl describe prometheusrule kubernetes-alerts -n monitoring
```

Application alerts:

```bash
kubectl describe prometheusrule application-alerts -n monitoring
```

Confirm:

- Rule groups exist
- Expressions are valid
- Alert definitions are loaded successfully
- No parsing errors are reported

Once Prometheus loads the rules successfully, alerts become available to Alertmanager for routing.
